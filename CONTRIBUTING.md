# Contributing to Mars Bluetooth HCI

Thanks for contributing to `mars-bluetooth-hci` — the open-source interface to the
Metirionic Advanced Ranging Stack (MARS). This guide documents the development and
release workflow so you can run CI locally and cut a release without hitting hidden
failures.

Contributions are welcome via pull requests against `main`. By contributing you agree
that your changes are licensed under the [MIT License](LICENSE). Issues and PRDs live on
the [GitHub issue tracker](https://github.com/Metirionic/mars-bluetooth-hci/issues).

## Quick reference — before you push

```bash
export RUSTFLAGS="-D warnings"          # warnings are errors (same as CI)

cargo check   --all-targets
cargo test
cargo clippy  --all-targets
cargo +nightly fmt --all -- --check

# If you changed any public FFI type, also regenerate and commit the C header:
./mars-bluetooth-hci/generate_headers.sh mars-bluetooth-hci/mars_bluetooth_hci.h
```

The sections below explain each of these and the rest of the workflow.

## 1. The generated C header (CI header-regeneration contract)

The committed C header **[`mars-bluetooth-hci/mars_bluetooth_hci.h`](mars-bluetooth-hci/mars_bluetooth_hci.h)**
is *generated* by [`safer-ffi`](https://github.com/getditto/safer_ffi)'s `headers`
feature — it is not hand-written and must not be edited by hand (its banner reads
`File auto-generated by ::safer_ffi` / `Do not manually edit this file`).

**Regenerate it with:**

```bash
./mars-bluetooth-hci/generate_headers.sh mars-bluetooth-hci/mars_bluetooth_hci.h
```

This runs `cargo run --features headers --bin generate_bluetooth_hci_headers` on the
host (default features) and writes the header to the path you pass. Build artifacts go
to `target-headers/` so your normal `target/` stays clean. The `headers` feature is
host-only and is **not** part of `default`.

**CI verifies the header is up to date.** The `header-check` job in
[`.github/workflows/ci.yml`](.github/workflows/ci.yml) regenerates the header to
`/tmp/mars_bluetooth_hci.h` and runs `diff` against the committed copy — **any
difference fails CI**.

**Rule:** any change to a public FFI type — an item annotated `#[ffi_export]` or
`#[derive_ReprC]`, or anything else that affects the generated header — requires
regenerating and committing `mars_bluetooth_hci.h` in the same PR. A PR that changes the
FFI surface without regenerating the header will pass locally but fail the
`header-check` job.

## 2. Local development workflow

Mirror what CI runs ([`.github/workflows/ci.yml`](.github/workflows/ci.yml)). The
toolchain is **stable** Rust for `check`/`test`/`clippy` and **nightly** Rust + `rustfmt`
for formatting (both crates use `edition = "2024"`).

```bash
# Warnings are errors — CI sets RUSTFLAGS="-D warnings" globally (ci.yml).
export RUSTFLAGS="-D warnings"

cargo check   --all-targets
cargo test
cargo clippy  --all-targets
cargo +nightly fmt --all -- --check
```

- `cargo +nightly fmt --all` (without `--check`) applies formatting.
- Nightly `rustfmt` is required because [`rustfmt.toml`](rustfmt.toml) uses nightly-only
  options: `group_imports = "StdExternalCrate"`, `imports_granularity = "Module"`,
  `max_width = 120`.
- One-time setup:
  ```bash
  rustup toolchain install nightly
  rustup component add rustfmt --toolchain nightly
  ```

These four commands cover CI's build/test/lint/format jobs; CI also runs the
version-literal check in §5. Passing all four locally (plus the header check in
§1 and the version-literal check in §5) is equivalent to passing CI.

## 3. `no_std` / embedded cross-compile testing

> **CI does not currently cross-compile.** Embedded builds are not gated on PRs, so this
> is a **manual local check** — run it before pushing changes to the FFI surface or
> feature set.

`no_std` is conditional:
`#![cfg_attr(all(not(test), not(feature = "std")), no_std)]`
([`mars-bluetooth-hci/src/lib.rs`](mars-bluetooth-hci/src/lib.rs), and identically in
`mars-common/src/lib.rs`) — `no_std` is active when the `std` feature is off and it is
not a test build.

The target triple is **not pinned** by this repo — the consumer chooses it via the CMake
variable `${RUST_TARGET}` (see
[`mars-bluetooth-hci/mars-bluetooth-hci-rust-config.cmake`](mars-bluetooth-hci/mars-bluetooth-hci-rust-config.cmake)).
The example below uses `thumbv6m-none-eabi` (the target shown in
[`mars-bluetooth-hci/README.md`](mars-bluetooth-hci/README.md)); substitute your own
target as needed.

```bash
rustup target add thumbv6m-none-eabi

# mars-bluetooth-hci — the embedded feature set straight from the CMake config.
# Use --lib: the generate_bluetooth_hci_headers bin target needs std (it is a
# host-only header generator), so only the library is no_std-compatible.
cargo check --lib --target thumbv6m-none-eabi \
  --no-default-features --features libc,alloc,libc-panic,libc-alloc

# mars-common — the matching feature subset (from its README snippet)
cargo check -p mars-common --lib --target thumbv6m-none-eabi \
  --no-default-features --features libc,libc-alloc,libc-panic
```

The `--lib` flag is required for `mars-bluetooth-hci`: `cargo check` without it also
checks the `generate_bluetooth_hci_headers` bin, which uses `std` and cannot compile
under `--no-default-features`. The CMake config uses `cargo build --lib …` for the same
reason. `mars-common` has no bin target, so `--lib` there is just for consistency.

**Feature matrix** (from
[`mars-bluetooth-hci/Cargo.toml`](mars-bluetooth-hci/Cargo.toml) and
`mars-common/Cargo.toml`):

| Feature      | Effect                                                                                  |
| ------------ | --------------------------------------------------------------------------------------- |
| `libc`       | Enables the C FFI surface (`#[ffi_export]` items).                                      |
| `alloc`      | Enables `postcard`'s `alloc` feature.                                                   |
| `libc-alloc` | Installs a `malloc`/`free` global allocator (`mars-common/src/libc/alloc.rs`).          |
| `libc-panic` | Installs a panic handler that calls a C-provided `rust_panic_cb` (`mars-common/src/libc/panic.rs`). |

`libc-alloc` and `libc-panic` both imply `libc`, so listing `libc` explicitly in the set
is harmless and is the convention used by the CMake config and the README snippets.

Embedded builds require `panic = "abort"` in your Cargo profile (per the crate README).
The CMake config builds the `staticlib` (`libmars_bluetooth_hci.a`) with exactly the
`--no-default-features --features libc,alloc,libc-panic,libc-alloc` set when `HOST` is
off, and exposes an imported CMake target `mars_bt_hci_rust`. For the full integrator
walkthrough (building/linking the static library, the FFI call pattern, a C example), see
the [C/embedded integration guide](docs/c-embedded-integration.md) (#14).

## 4. Releases (`cog` / Conventional Commits)

This repo uses [Conventional Commits](https://www.conventionalcommits.org/) enforced by
[cocogitto](https://github.com/cocogitto/cocogitto) (`cog`); see
[`cog.toml`](cog.toml). Use standard commit types (`feat:`, `fix:`, …). The `chore`,
`ci`, and `test` types are omitted from the changelog. No commit scopes are restricted.

**Releases are automatic.** You do **not** run `cog bump` locally. On every push to
`main`, the [Release workflow](.github/workflows/release.yml) runs `cog bump --auto
--dry-run` to detect whether a version bump is needed; if so, it runs `cog bump --auto`,
which:

1. bumps the affected package(s) and writes the new version to each crate's `Cargo.toml`
   (via the `cargo set-version` `pre_package_bump_hook`),
2. creates a version-bump commit (`chore(version): <crate>@<X.Y.Z>`),
3. creates a per-package tag `<crate>@<X.Y.Z>`,
4. regenerates each package's `CHANGELOG.md`,
5. pushes the commit and tags.

So to cut a release, **just merge a `feat:` or `fix:` commit to `main`** and the workflow
does the rest.

- **Tag format:** `<crate>@<X.Y.Z>` — e.g. `mars-bluetooth-hci@0.8.0`, `mars-common@0.2.0`
  (`generate_mono_repository_package_tags = true`, `monorepo_version_separator = "@"`).
  There is no global/workspace tag.
- **Bump order:** `mars-common` (`bump_order = 1`) is bumped before `mars-bluetooth-hci`
  (`bump_order = 2`), matching the dependency direction.
- **crates.io publishing is manual.** There is no `cargo publish` step in CI. After the
  bump commit and tags land on `main`, a maintainer publishes in bump order:
  ```bash
  cargo publish -p mars-common
  cargo publish -p mars-bluetooth-hci
  ```
  No GitHub Release objects are created (that job was removed). Releases only happen from
  `main` (`branch_whitelist = ["main"]`).

## 5. Keeping README and docs/ version literals in sync

The per-crate READMEs each contain a dependency snippet with a version literal, and
`docs/` prose names the crates with version literals; both must track the published
crate version:

- [`mars-bluetooth-hci/README.md`](mars-bluetooth-hci/README.md) —
  `mars-bluetooth-hci = { version = "…", … }`
- [`mars-common/README.md`](mars-common/README.md) —
  `mars-common = { version = "…", … }`
- [`docs/architecture.md`](docs/architecture.md) — `mars-bluetooth-hci` and
  `mars-common` in the `(vX.Y.Z)` prose form
- [`docs/c-embedded-integration.md`](docs/c-embedded-integration.md) —
  `mars-bluetooth-hci` in the `@X.Y.Z` prose form (`GIT_TAG` pin)

The `cog` `cargo set-version` hook only updates `Cargo.toml`, not the READMEs or the
docs, so updating every version literal is still a **manual pre-release step** — after
a version bump, update each literal to match the new `Cargo.toml` version. A CI gate
catches drift: the `version-check` job in
[`.github/workflows/ci.yml`](.github/workflows/ci.yml), backed by
[`.github/scripts/check_readme_versions.py`](.github/scripts/check_readme_versions.py),
walks every crate in `[workspace].members` and fails the build if either a sub-README's
`<crate> = { version = "…", … }` snippet diverges from the crate's `Cargo.toml`
`version` (both compared on `MAJOR.MINOR`, since the snippets intentionally use a
minor-only SemVer requirement), or a `docs/` prose literal of the `(vX.Y.Z)` or
`@X.Y.Z` form diverges (compared on the full version, so a patch bump must update the
prose too). Run it locally before pushing:

```bash
python3 .github/scripts/check_readme_versions.py
```

The root [`README.md`](README.md) is not a workspace-member crate README and carries no
version literals, so it needs no syncing. The cog-generated `*/CHANGELOG.md` files are also out
of scope.

## 6. Documentation structure

This repo uses a hybrid documentation layout (see PRD #3 and
[`docs/agents/domain.md`](docs/agents/domain.md)):

- **Root meta files:** [`README.md`](README.md) (front door), [`CONTEXT.md`](CONTEXT.md) (concise
  agent-facing context — #13), this `CONTRIBUTING.md`, and [`LICENSE`](LICENSE).
- **[`docs/`](docs) tree** for technical, human-facing documents: currently
  [`docs/ecosystem.md`](docs/ecosystem.md), [`docs/architecture.md`](docs/architecture.md),
  [`docs/wire-format.md`](docs/wire-format.md), [`docs/c-embedded-integration.md`](docs/c-embedded-integration.md)
  (the C/embedded integration guide, #14), and [`docs/agents/`](docs/agents).
- **[`docs/adr/`](docs/adr)** for Architecture Decision Records, seeded by #6
  (ADR-0001 wire format, ADR-0002 serialize-only FFI). New ADRs are named
  `docs/adr/NNNN-kebab-title.md`.

**Where to add new docs:** technical content → `docs/`; architecture decisions →
`docs/adr/`; concise agent context → `CONTEXT.md`. Use **Mermaid** for diagrams (GitHub
renders it natively, matching the sibling `mars-cs-nrf54l` repo). **Cross-link rather
than duplicate** — for example, point at the sibling firmware repo's architecture doc
instead of restating it. Use the glossary vocabulary from [`CONTEXT.md`](CONTEXT.md); if
a concept is missing, note the gap rather than inventing a new term.
